# Codex 文档格式执行规范

本文件用于约束 Codex 在本仓库或引用本 Skill 的项目中处理文档类任务时的行为。

## 1. 总体原则

- 处理文档时，优先遵守 `SKILL.md` 中的文档格式与 Markdown 转 DOCX 规范。
- 不得把 Markdown 源文档原样塞入 DOCX。
- 不得在 Word 正文区残留未处理的 Markdown 标记符号。
- 代码块属于保护区，代码块内的 Markdown 示例不得被误解析为正文标题、正文表格或正文链接。
- 不得因为格式清洗改变原文事实、技术路线、章节含义和项目边界。
- 格式修改和内容改写必须分开处理。
- 长文档修改时，不要整篇打印到聊天界面，应输出修改摘要、问题清单和生成文件路径。

## 2. Markdown 转 DOCX 必做流程

当任务涉及 Markdown 转 Word/DOCX 时，必须按以下顺序执行：

1. 读取 Markdown 源文档。
2. 识别代码围栏范围，将代码块标记为保护区。
3. 识别标题、正文、列表、表格、代码块、图片和链接。
4. 判断标题映射关系：通常 `#` 为文档标题，`##` 为一级标题，`###` 为二级标题，`####` 为三级标题。
5. 检查标题是否已有手工编号；如果已有编号，不得再叠加 Word 自动编号。
6. 清理正文区 Markdown 残留符号。
7. 将标题映射为 Word 标题样式。
8. 将 Markdown 表格转换为真实 Word 表格。
9. 将代码块转换为等宽字体段落，不保留反引号围栏。
10. 将 `---`、`***`、`___` 等分割线删除或转换为分页/段落分隔。
11. 将 `**模块定位**：`、`**输入数据**：`、`**处理流程**：` 等加粗标签转换为 Word 加粗标签。
12. 将 `表1`、`图1` 规范为“表 1”“图 1”。
13. 将表格单元格中的 `<br>` 转换为单元格内换行。
14. 统一正文缩进、行距、段前段后和列表编号。
15. 设置页边距、页码、必要目录、图表编号。
16. 生成 DOCX 后执行残留检查和交付前验收。
17. 输出修改摘要和交付文件路径。

## 3. DOCX 残留检查清单

生成 DOCX 后必须检查正文区是否仍包含以下内容：

- `#`、`##`、`###`、`####` 标题残留。
- `**`、`__`、`~~` 等强调标记。
- 单独一行 `---`、`***`、`___`。
- 三个反引号代码围栏。
- Markdown 表格管道符 `|`。
- Markdown 链接格式 `[标题](链接)`。
- Markdown 图片格式 `![说明](路径)`。
- HTML 换行标签 `<br>`、`<br/>`、`<br />`。
- 多余转义符 `\*`、`\#`、`\_`。

说明：代码块区可以保留示例内容中的 `#`、`|`、`[标题](链接)` 等字符，但必须使用代码块样式呈现，且不得保留反引号围栏。

## 4. Word 排版硬性要求

- 页面：A4，纵向。
- 页边距：上 2.5 cm、下 2.5 cm、左 2.8 cm、右 2.8 cm。
- 文档标题：黑体，二号或小二号，居中，加粗。
- 一级标题：黑体，三号，居左，加粗。
- 二级标题：黑体，小三号，居左，加粗。
- 三级标题：黑体，四号，居左，加粗。
- 正文：宋体或仿宋，四号或小四号，两端对齐，首行缩进 2 字符。
- 正文行距：1.5 倍行距或固定值 28 磅，全文统一。
- 正文段前：0 磅。
- 正文段后：0 磅或 6 磅，全文统一。
- 表格文字：宋体或仿宋，五号。
- 代码块：Consolas 或 Courier New，五号。
- 正文字体颜色统一为黑色。

## 5. 编号与列表要求

- 标题编号使用：1、1.1、1.1.1。
- 如果 Markdown 标题已经包含编号，不得再叠加自动编号。
- 正文第一层列举使用：（1）（2）（3）。
- 正文第二层列举使用：1）2）3）。
- 正文第三层列举使用：a）b）c）。
- 同一层级不得混用短横线、圆点、①、1）、一是等格式。
- Markdown 无序列表转 DOCX 时，不保留 `-`、`*`、`+` 原始符号。

## 6. 目录、页码、图表要求

- 正式 DOCX 建议生成自动目录。
- 目录只包含一级至三级标题。
- 目录不得包含代码块中的 Markdown 标题示例。
- 不得手工输入目录页码。
- 正文首页页码从 1 开始，页码位于页脚居中。
- 表题置于表格上方，例如“表 1 模块功能清单”。
- 图题置于图片下方，例如“图 1 系统总体架构”。
- 正文引用图表时使用“如图 1 所示”“见表 2”，不使用“上图”“下表”。

## 7. 表格与代码块要求

- Markdown 表格必须转换为真实 Word 表格。
- 表格内的 `<br>` 必须转换为单元格内换行。
- 表格不得超出页面边距。
- 长表格跨页时应重复表头。
- 代码块必须删除反引号围栏，但保留代码内容和语言语义。
- `json`、`sql`、`http`、`markdown`、`text` 等代码块均按代码样式处理。
- 超过 30 行的代码块或目录树，可放入附录或保持紧凑代码块。

## 8. 内容审查要求

处理技术方案、设计文档、模块说明时，应检查：

- 是否说明模块定位、输入数据、处理流程、输出结果、异常处理、验收标准。
- 是否存在空泛表述、营销语言、AI 化套话。
- 是否将未实现能力写成已完成能力。
- 是否编造未确认的技术指标、性能指标、模型能力或数据规模。
- 是否明确开源项目的使用方式：直接复用、接口调用、数据源接入或架构参考。

## 9. 交付前验收清单

交付 DOCX 前必须检查：

1. 正文区是否无 Markdown 残留。
2. 代码块区是否保留为代码样式，且未被误解析为正文标题或表格。
3. 标题映射是否正确：`#` 文档标题、`##` 一级标题、`###` 二级标题、`####` 三级标题。
4. 标题编号是否连续，是否避免重复编号。
5. 正文是否两端对齐、首行缩进 2 字符。
6. 正文行距、段前段后是否统一。
7. 正文列表编号是否统一。
8. 表格是否为真实 Word 表格，是否超出页边距。
9. 表格中的 `<br>` 是否转换为单元格内换行。
10. 长表格跨页时是否重复表头。
11. 图题、表题编号是否连续。
12. 是否存在大段空白页、孤立标题、异常分页。
13. 是否存在明显 AI 化套话和空泛描述。

## 10. 输出要求

- 简单修改可以直接说明修改点。
- 长文档修改必须生成文件，不要在对话中整篇输出。
- 审查类任务优先输出编号清单。
- 转换类任务必须说明是否完成 Markdown 残留清理和 DOCX 交付前验收。
