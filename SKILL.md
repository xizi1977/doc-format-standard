# 文档格式与 Markdown 转 DOCX 规范

当被要求生成文档、技术方案、模块说明、清洗 Markdown 或将 Markdown 转为 DOCX/Word 文档时，必须严格遵守以下规范。

本 Skill 的核心目标不是让文档看起来复杂，而是让文档具备以下特征：

- Markdown 源文件结构清晰，便于版本管理。
- DOCX 文件中不得残留未处理的 Markdown 标记符号。
- Word 排版统一、正式、可评审、可归档。
- 内容表达工程化、可执行、可验收，避免明显 AI 化套话。

## 1. 任务识别与处理原则

处理文档前，先判断任务类型：

- 生成新文档：按本规范直接生成结构化内容。
- 清洗 Markdown：先清理 Markdown 残留，再输出规范化 Markdown 或 DOCX。
- Markdown 转 DOCX：必须执行“结构识别、残留清理、样式映射、成品验收”四步。
- 审查已有 DOCX：优先输出问题清单、修改建议和缺项，不直接改动事实内容。
- 优化已有文档：格式修改和内容改写分开处理，避免因排版清洗改变正文含义。

不得擅自改变文档事实、技术路线、章节含义、项目边界、指标口径和交付范围。

## 2. Markdown 结构识别原则

Markdown 转 DOCX 前，必须先识别结构，不能直接逐行替换符号。

### 2.1 代码围栏保护区

代码围栏内的内容属于“示例代码/示例文本保护区”，不得按正文 Markdown 解析。

要求：

- 代码围栏外的 `#`、`**`、`---`、`|` 等符号需要按正文规则清洗。
- 代码围栏内的 `#`、`**`、`|`、`[标题](链接)` 等符号可以作为代码示例保留，但必须使用代码块样式呈现。
- 代码围栏本身必须删除，不得在 DOCX 中保留三个反引号。
- 残留检查应区分正文区和代码块区，不得把代码示例中的 `#` 误判为正文标题残留。
- 如果代码块语言为 markdown，说明其内容是 Markdown 示例，应整体作为代码块保留，不得拆成 Word 标题和正文。

### 2.2 Markdown 标题到 Word 样式映射

常见技术文档会使用 `#` 作为文档标题，`##` 作为一级章节，`###` 作为二级章节，`####` 作为三级章节。转换时必须先判断标题层级含义。

推荐映射：

| Markdown 标题 | Word 中的处理方式 |
|---|---|
| `# 文档名称` | 文档标题，不进入正文标题编号体系 |
| `## 1 一级章节` | 一级标题 |
| `### 1.1 二级章节` | 二级标题 |
| `#### 1.1.1 三级章节` | 三级标题 |
| `#####` 及更深层级 | 原则上转为正文加粗小标题或正文编号列表，不再作为 Word 标题 |

要求：

- 不得简单地把 `#` 直接转换成一级标题，否则会导致文档标题和正文一级标题混乱。
- 如果标题文本已经包含 `1`、`1.1`、`1.1.1` 等编号，不得再叠加 Word 自动编号，避免出现“1 1 项目定位”这类重复编号。
- 如果启用 Word 自动编号，应先去除 Markdown 标题中的手工编号；如果保留手工编号，则不要再套自动编号。
- `#### 1.1.1` 在技术文档中常代表三级业务标题，不应简单降级为普通正文。

### 2.3 加粗标签与字段说明识别

Markdown 中常见以下写法：

- `**表1 第一版功能清单**`
- `**模块定位**：……`
- `**输入数据**：……`
- `**处理流程**：`
- `**验收标准**：……`

转换要求：

- `**表1 ...**`、`**表 1 ...**` 应识别为表题，放在表格上方，规范为“表 1 标题”。
- `**图1 ...**`、`**图 1 ...**` 应识别为图题，放在图片下方，规范为“图 1 标题”。
- `**模块定位**：正文` 应转换为“模块定位：”加粗，后续正文为普通字体，不保留 `**`。
- `**处理流程**：` 这类单独标签应转换为加粗小标题或标签段落，下方列表继续作为正文流程。
- 不得把所有加粗文本都删除样式；应保留必要的标签加粗效果，但去除 Markdown 符号。

## 3. 页面与版式要求

- 页面采用 A4 纸张，纵向。
- 页边距：上 2.5 cm、下 2.5 cm、左 2.8 cm、右 2.8 cm。
- 正文字体颜色统一为黑色，不得使用蓝色、红色、绿色等彩色字体突出内容。
- 不使用艺术字、阴影、装饰性符号、表情符号、复杂渐变背景。
- 不得通过连续空格、Tab、多余空行制造缩进或段落间距。
- 技术方案正文原则上采用“文字说明 + 表格 + 流程步骤 + 必要代码片段”的形式。

## 4. 字体与字号要求

- 文档标题：黑体，二号或小二号，居中，加粗。
- 一级标题：黑体，三号，居左，加粗。
- 二级标题：黑体，小三号，居左，加粗。
- 三级标题：黑体，四号，居左，加粗。
- 正文：宋体或仿宋，四号或小四号。
- 表格文字：宋体或仿宋，五号。
- 代码块、接口字段、数据库字段：Consolas 或 Courier New，五号。
- 图题、表题：宋体或黑体，小四号，加粗，编号连续。

## 5. 正文段落与缩进规范

普通正文必须统一设置段落格式：

- 对齐方式：两端对齐。
- 首行缩进：2 字符。
- 行距：优先使用 1.5 倍行距；如需固定值，可使用 28 磅，但全文必须统一。
- 段前：0 磅。
- 段后：0 磅或 6 磅，全文统一。
- 标题、表格、图题、表题、代码块不使用正文首行缩进。
- 不允许用空行模拟段落间距。
- 不允许用连续空格或 Tab 模拟首行缩进。
- 不允许出现大段空白、孤立标题、标题位于页面底部但正文转到下一页的情况。

## 6. 标题与编号规范

标题编号采用统一层级格式：

- 一级标题：1、2、3。
- 二级标题：1.1、1.2、1.3。
- 三级标题：1.1.1、1.1.2、1.1.3。

要求：

- 不得混用“一、”“（一）”“1）”“①”等多套标题编号体系。
- 同一级标题的表达方式应保持一致。
- 标题应具体明确，避免使用“总体介绍”“相关内容”“能力说明”等空泛标题。
- 不建议超过三级标题；确需更深层级时，改用正文编号列表或正文加粗小标题。

## 7. 正文序号与列表规范

正文内部列举与标题编号必须区分。

推荐格式：

| 使用场景 | 推荐格式 |
|---|---|
| 标题编号 | 1、1.1、1.1.1 |
| 正文第一层列举 | （1）（2）（3） |
| 正文第二层列举 | 1）2）3） |
| 正文第三层列举 | a）b）c） |
| 表格内简单列举 | 1、2、3 |

要求：

- 同一层级不得混用短横线、圆点、①、1）、一是等格式。
- Markdown 无序列表转 DOCX 时，默认转换为正文编号列表或规范项目符号，不保留原始 `-`、`*`、`+`。
- 正式技术文档中不建议大量使用圆点项目符号，优先使用（1）（2）（3）。
- 列表缩进应统一，不能出现同级列表左右不齐。
- 表格内不使用复杂多级编号。

## 8. Markdown 转 DOCX 前置清洗要求

大部分文档交付场景是“先生成 Markdown，再转换为 DOCX”。因此，在将 Markdown 转为 DOCX 之前，必须先执行格式清洗，不得把 Markdown 原文直接写入 Word。

### 8.1 禁止残留的 Markdown 符号

DOCX 正文区不得出现以下 Markdown 残留：

- 标题符号：`#`、`##`、`###`、`####`。
- 加粗符号：`**文字**`、`__文字__`。
- 斜体符号：`*文字*`、`_文字_`。
- 删除线符号：`~~文字~~`。
- 分割线符号：`---`、`***`、`___`。
- 引用符号：`>`。
- 无序列表残留：多余的 `-`、`*`、`+`。
- 代码围栏：三个反引号及其后的语言标识，如 text、json、sql、python、markdown。
- 行内代码反引号：例如 `字段名`、`接口名`。
- Markdown 链接原始格式：`[标题](链接)`。
- 图片原始格式：`![说明](路径)`。
- 表格管道符残留：`| 字段 | 说明 |`。
- HTML 换行残留：`<br>`、`<br/>`、`<br />`。
- 多余转义符：`\*`、`\#`、`\_`。

说明：代码块区可以保留示例内容中的 `#`、`|`、`[标题](链接)` 等字符，但必须以代码块样式呈现，不能混入普通正文。

### 8.2 Markdown 标记转换规则

清洗时应按照以下规则转换，而不是简单删除：

| Markdown 内容 | DOCX 中应转换为 |
|---|---|
| `# 文档名称` | 文档标题 |
| `## 1 标题` | 一级标题 |
| `### 1.1 标题` | 二级标题 |
| `#### 1.1.1 标题` | 三级标题 |
| `**重点内容**` | 保留文字，按需加粗，不保留 `**` |
| `**模块定位**：正文` | “模块定位：”加粗，正文普通字体 |
| `**表1 标题**` | “表 1 标题”表题 |
| `---` | 删除，或转换为分页/段落分隔，不保留横线字符 |
| `- 条目` | Word 项目符号或正文编号列表 |
| `1. 条目` | Word 编号列表或普通编号段落 |
| 代码围栏 | 等宽字体代码段，不保留围栏符号 |
| `[标题](链接)` | 显示为“标题”，必要时在括号或脚注中保留链接 |
| `![说明](路径)` | 插入图片并保留图题，不保留 Markdown 图片语法 |
| Markdown 表格 | 转为 Word 表格，不保留管道符 |
| `<br>` | 表格单元格内换行或段内换行，不保留 HTML 标签 |

### 8.3 长横线与分割线处理

Markdown 中的 `---`、`***`、`___` 在 Word 中经常变成无意义长横线。处理要求如下：

- 用于章节分割时，应删除分割线，改用标题层级和段前段后间距体现结构。
- 用于强调时，应改写为普通段落，不保留横线。
- 用于页面分隔时，应转换为 Word 分页符，而不是三条短横线。
- DOCX 正文中不得出现单独一行 `---`、`***`、`___`。

## 9. 代码块、接口与配置示例规范

- 代码块是保护区，代码块内部内容不得按普通 Markdown 正文解析。
- 删除代码块开头和结尾的反引号围栏。
- 保留必要代码内容，使用 Consolas 或 Courier New，五号。
- 代码块前必须有一句说明其用途。
- 非必要代码不得大量堆放。
- JSON、SQL、YAML、Python、HTTP、Markdown 示例必须保持原有语义和格式。
- 行内代码标记转换为普通文本或等宽字体，不保留反引号。
- 接口名、字段名、数据库表名、配置项名称保持原样，不强行翻译。
- 超过 30 行的长代码块、目录树、长 SQL 示例，可移动到附录或保留为紧凑代码块，避免正文分页严重破碎。

## 10. 表格规范

- Markdown 表格转 DOCX 时必须转换为真实 Word 表格，不得保留管道符。
- 表题置于表格上方，例如“表 1 模块功能清单”。
- 如果 Markdown 中写作“表1”，转换时统一规范为“表 1”。
- 表头加粗，表格文字五号。
- 表格宽度应适应页面，不得超出页边距。
- 长表格跨页时应重复表头。
- 单元格上下左右边距保持一致。
- 表格列名应统一，常用列名为“字段名、类型、是否必填、说明”或“模块、功能、输入、输出、验收标准”。
- 单元格中的 `<br>` 必须转换为单元格内换行，不得原样保留 HTML 标签。
- 内容较多时优先调整列宽和换行，不得把表格转换成截图。

## 11. 图示、图题与引用规范

- 技术方案可加入：系统总体架构图、数据处理流程图、小智能体任务流转图、数据库核心表关系图、领导演示流程图。
- 图示风格应为黑白线框图，白底黑字，结构清晰，箭头方向明确。
- 图中不使用卡通图标、复杂渐变、装饰性背景。
- 图中文字应与正文术语保持一致。
- 图题置于图片下方，例如“图 1 系统总体架构”。
- 如果 Markdown 中写作“图1”，转换时统一规范为“图 1”。
- 图表编号全文连续，例如图 1、图 2、表 1、表 2。
- 正文引用图表时使用明确编号，例如“如图 1 所示”“见表 2”，不使用“上图”“下表”等相对表述。

## 12. 目录、页眉页脚与页码规范

正式 DOCX 文档建议包含自动目录。

- 目录置于封面之后、正文之前。
- 目录只包含一级至三级标题，不包含文档标题和代码块中的 Markdown 标题示例。
- 不得手工输入目录页码。
- 交付前必须更新目录域，确保标题文字和页码与正文一致。
- 正式 DOCX 应设置页码，页码位于页脚居中。
- 正文首页页码从 1 开始。
- 除非用户明确要求，不添加复杂页眉、水印、彩色页脚或装饰线。
- 页眉可填写文档简称，字体小五号，颜色黑色或深灰。

## 13. 封面、版本记录与交付信息

正式交付文档建议包含封面和版本记录表。

封面至少包含：

- 文档名称。
- 版本号。
- 编制单位。
- 编制日期。

版本记录表至少包含：

- 版本号。
- 修订日期。
- 修订说明。
- 修订人。

如用户未提供编制单位、修订人等信息，不得编造，可保留为空或标注“待补充”。

## 14. 中英文、数字与标点规范

- 中文正文优先使用中文标点。
- 英文缩写、接口名、字段名、版本号、端口号保持原样。
- CVE、NVD、API、DOCX、JSON、SQL、HTTP、Markdown 等缩写保持原样。
- 中英文、中文与数字之间的空格策略应全文一致。
- 中文正文优先使用中文引号“”。
- 中文说明使用中文冒号：。
- 直引号、半角冒号等如属于代码、接口、JSON、SQL、HTTP 示例，应保持原样。
- 不得同一文档内混用全角和半角标点造成风格不一致。

## 15. 正文写作要求

文档面向开发人员使用时，应突出“怎么做、输入是什么、输出是什么、边界是什么、如何验收”。

模块描述建议采用以下结构：

- 模块定位：该模块的作用与核心价值。
- 输入数据：依赖的前置条件或数据来源。
- 处理流程：具体的业务逻辑或技术实现步骤。
- 输出结果：模块执行完毕后的产物。
- 异常处理：遇到错误时的应对机制。
- 验收标准：如何验证该模块功能正常。

要求：

- 不得编造未确定的技术指标、性能指标、模型能力或数据规模。
- 对暂未实现的能力，应明确标注为“后续扩展”，不得写成已经完成。
- 对开源项目的使用方式应写清楚是“直接复用”“接口调用”“数据源接入”还是“架构参考”。

## 16. 去 AI 化行文要求

- 减少模板化套话，避免频繁使用“通过……实现……提升……”的句式。
- 每段文字应有明确的信息增量，不写没有实际内容的过渡段。
- 技术描述应绑定具体对象，例如数据表、接口、任务、字段、配置文件、页面功能。
- 不使用“全方位、立体化、智能化、体系化、闭环化”等宏大词堆叠。
- 不使用“领先、先进、强大、极大提升、全面赋能”等营销语言，除非有明确依据。
- 不使用“这个模块很重要”“效果很好”“跑起来就行”等口语化表达。
- 每个功能点应能被开发人员转化为代码任务或测试用例。
- 所有结论性描述应尽量配套依据，例如数据来源、字段规则、接口返回、验收方式。

## 17. Codex 执行要求

当 Codex 使用本规范处理文档时，应按以下顺序执行：

1. 识别任务类型：生成新文档、清洗 Markdown、转换 DOCX、审查已有文档。
2. 先识别代码围栏范围，将代码块标记为保护区。
3. 识别 Markdown 标题层级映射关系，确认 `#` 是文档标题还是正文标题。
4. 如果标题已经包含手工编号，禁止再叠加自动编号。
5. 清理 Markdown 残留，再进行 Word 排版。
6. 将加粗标签、表题、图题、字段说明转换为 Word 样式，不保留 Markdown 符号。
7. 将 `<br>` 转换为换行，不保留 HTML 标签。
8. 如果输出 DOCX，必须检查正文区是否仍包含 `#`、`**`、反引号围栏、`---`、Markdown 表格管道符等残留。
9. 对长文档不要整篇打印到聊天界面，应生成文件或只输出问题清单、修改摘要和交付路径。
10. 不得擅自改变文档事实、技术路线、章节含义和项目边界。
11. 格式修改和内容改写应分开处理，避免因排版清洗导致正文含义变化。
12. 生成 DOCX 后必须执行一次交付前检查。

## 18. DOCX 交付前验收清单

交付前必须逐项检查：

1. 正文区是否无 `#`、`**`、`---`、反引号围栏、`|` 等 Markdown 残留。
2. 代码块区是否保留为代码样式，未被误解析为正文标题或表格。
3. `#` 是否已转换为文档标题，`##/###/####` 是否正确映射为一级至三级标题。
4. 标题编号是否连续，是否避免了重复编号。
5. 正文是否两端对齐、首行缩进 2 字符。
6. 正文行距、段前段后是否统一。
7. 正文列表编号是否统一，是否存在同层级混用。
8. `**模块定位**：` 等标签是否转成 Word 加粗标签而非残留 Markdown。
9. `表1`、`图1` 是否规范为“表 1”“图 1”。
10. 目录页码是否更新。
11. 页码是否正常。
12. 表格是否为真实 Word 表格，是否超出页边距。
13. 表格中的 `<br>` 是否转换为单元格内换行。
14. 长表格跨页时是否重复表头。
15. 图题、表题编号是否连续，引用是否明确。
16. 是否存在大段空白页、孤立标题、异常分页。
17. 是否存在明显 AI 化套话和空泛描述。
18. 是否将未实现能力写成已完成能力。
19. Markdown、DOCX、PDF 三个版本的标题、章节编号、图表编号是否一致。

## 19. 交付要求

- Markdown 源文件用于版本管理和开发协作。
- DOCX 文件用于内部评审和领导汇报。
- PDF 文件用于正式发送和归档。
- 三个版本的标题、章节编号、表格编号和图编号应保持一致。
- 交付前必须完成残留检查、格式检查和内容审查。
