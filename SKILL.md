# 文档格式与 Markdown 转 DOCX 规范

当被要求生成文档、技术方案、模块说明、清洗 Markdown 或将 Markdown 转为 DOCX/Word 文档时，必须严格遵守以下规范。

本 Skill 的核心目标不是让文档看起来复杂，而是让文档具备以下特征：

- Markdown 源文件结构清晰，便于版本管理。
- DOCX 文件中不得残留 Markdown 标记符号。
- Word 排版统一、正式、可评审、可归档。
- 内容表达工程化、可执行、可验收，避免明显 AI 化套话。

## 1. 页面与版式要求

- 文档页面采用 A4 纸张，页面方向为**纵向**，页边距：上 2.5 cm、下 2.5 cm、左 2.8 cm、右 2.8 cm。
- 正文字体颜色统一为**黑色**，不得使用蓝色、红色、绿色等彩色字体突出内容。
- 全文不得随意使用加粗、斜体、下划线、阴影、艺术字、表情符号或装饰性符号。
- 段落之间保持统一间距，不得出现大段空白、行距忽大忽小、缩进不一致等问题。
- 技术方案正文原则上采用“文字说明 + 表格 + 流程步骤 + 必要代码片段”的形式，不使用过度花哨的排版。

## 2. 字体与字号要求

- 文档标题：黑体，二号或小二号，居中，加粗。
- 一级标题：黑体，三号，居左，加粗。
- 二级标题：黑体，小三号，居左，加粗。
- 三级标题：黑体，四号，居左，加粗。
- 正文：宋体或仿宋，四号或小四号，**首行缩进 2 字符**。
- 表格文字：宋体或仿宋，五号，内容上下居中、左右居中或左对齐保持一致。
- 代码块、接口字段、数据库字段：建议使用等宽字体，如 Consolas 或 Courier New，字号五号。
- 图题、表题：宋体或黑体，小四号，加粗，编号连续，例如“图 1 系统总体架构”“表 1 模块功能清单”。

## 3. Markdown 转 DOCX 前置清洗要求

大部分文档交付场景是“先生成 Markdown，再转换为 DOCX”。因此，在将 Markdown 转为 DOCX 之前，必须先执行格式清洗，不得把 Markdown 原文直接写入 Word。

### 3.1 禁止残留的 Markdown 符号

DOCX 正文中不得出现以下 Markdown 残留：

- 标题符号：`#`、`##`、`###`。
- 加粗符号：`**文字**`、`__文字__`。
- 斜体符号：`*文字*`、`_文字_`。
- 删除线符号：`~~文字~~`。
- 分割线符号：`---`、`***`、`___`。
- 引用符号：`>`。
- 无序列表残留：多余的 `-`、`*`、`+`。
- 代码围栏：三个反引号及其后的语言标识，如 text、json、sql、python。
- 行内代码反引号：例如 `字段名`、`接口名`。
- Markdown 链接原始格式：`[标题](链接)`。
- 图片原始格式：`![说明](路径)`。
- 表格管道符残留：`| 字段 | 说明 |`。
- 多余转义符：`\*`、`\#`、`\_`。

### 3.2 Markdown 标记转换规则

清洗时应按照以下规则转换，而不是简单删除：

| Markdown 内容 | DOCX 中应转换为 |
|---|---|
| `# 标题` | 一级标题样式 |
| `## 标题` | 二级标题样式 |
| `### 标题` | 三级标题样式 |
| `**重点内容**` | 保留文字，按需加粗，不保留 `**` |
| `---` | 删除，或转换为分页/段落分隔，不保留横线字符 |
| `- 条目` | Word 项目符号或普通段落 |
| `1. 条目` | Word 编号列表或普通编号段落 |
| 代码围栏 | 等宽字体代码段，不保留围栏符号 |
| `[标题](链接)` | 显示为“标题”，必要时在括号或脚注中保留链接 |
| `![说明](路径)` | 插入图片并保留图题，不保留 Markdown 图片语法 |
| Markdown 表格 | 转为 Word 表格，不保留管道符 |

### 3.3 长横线与分割线处理

Markdown 中的 `---`、`***`、`___` 在 Word 中经常变成无意义长横线。处理要求如下：

- 用于章节分割时，应删除分割线，改用标题层级和段前段后间距体现结构。
- 用于强调时，应改写为普通段落，不保留横线。
- 用于页面分隔时，应转换为 Word 分页符，而不是三条短横线。
- DOCX 正文中不得出现单独一行 `---`、`***`、`___`。

### 3.4 代码块处理

代码块不得原样保留 Markdown 围栏。处理要求如下：

- 删除代码块开头和结尾的反引号围栏。
- 保留代码内容本身。
- 代码字体使用 Consolas 或 Courier New，字号五号。
- 代码块前必须有一句说明其用途。
- 非必要代码不得大量堆放，配置、SQL、JSON 示例应保持格式正确。

### 3.5 表格处理

Markdown 表格转 DOCX 时必须转换为真实 Word 表格。处理要求如下：

- 删除 `|`、`---` 等 Markdown 表格语法。
- 表头加粗，表格文字五号。
- 表格应有编号和标题，例如“表 1 模块功能清单”。
- 表格列名统一，常用列名为“字段名、类型、是否必填、说明”或“模块、功能、输入、输出、验收标准”。

## 4. 标题与编号要求

- 标题编号采用统一层级格式：
  - 一级标题：1、2、3
  - 二级标题：1.1、1.2、1.3
  - 三级标题：1.1.1、1.1.2、1.1.3
- 不得混用“一、”“（一）”“1）”“①”等多套编号体系。
- 同一级标题的表达方式应保持一致，例如均采用“模块名称 + 说明”的形式。
- 标题应具体明确，避免使用“总体介绍”“相关内容”“能力说明”等过于空泛的标题。

## 5. 正文写作要求

- 文档面向开发人员使用，应突出“怎么做、输入是什么、输出是什么、边界是什么、如何验收”。
- **模块描述标准结构模板**（务必参考此结构进行描述）：
  - **模块定位**：该模块的作用与核心价值。
  - **输入数据**：依赖的前置条件或数据来源。
  - **处理流程**：具体的业务逻辑或技术实现步骤。
  - **输出结果**：模块执行完毕后的产物。
  - **异常处理**：遇到错误时的应对机制。
  - **验收标准**：如何验证该模块功能正常。
- 避免使用空泛表达（如“全面提升系统智能化水平”“构建先进完善的能力体系”“充分发挥人工智能赋能作用”“实现高质量发展”），应改为具体表达（如“系统每日定时拉取 CVE/NVD 数据，并写入 raw_vulnerabilities 表”“本地模型仅负责摘要、分类和字段提取，不作为漏洞真实性判断依据”“漏洞详情页应展示 CVE 编号、CVSS 分值、CWE 类型、影响产品、参考链接和模型生成摘要”）。
- 不得编造未确定的技术指标、性能指标、模型能力或数据规模。
- 对暂未实现的能力，应明确标注为“后续扩展”，不得写成已经完成。
- 对开源项目的使用方式应写清楚是“直接复用”“接口调用”“数据源接入”还是“架构参考”，避免表述含混。

## 6. 去 AI 化行文要求

- 减少模板化套话，避免频繁使用“通过……实现……提升……”的句式。
- 每段文字应有明确的信息增量，不写没有实际内容的过渡段。
- 技术描述应尽量绑定具体对象，例如数据表、接口、任务、字段、配置文件、页面功能。
- 不使用过度宏大的表述，例如“全方位、立体化、智能化、体系化、闭环化”堆叠。
- 不使用营销式语言，例如“领先”“先进”“强大”“极大提升”“全面赋能”等，除非有明确依据。
- 不使用口语化表达，例如“这个模块很重要”“效果很好”“跑起来就行”。
- 每个功能点应能被开发人员转化为代码任务或测试用例。
- 对模型生成内容必须增加约束：本地模型只能基于输入漏洞原文进行摘要、分类和字段提取，不得补充资料中不存在的事实。
- 所有结论性描述应尽量配套依据，例如数据来源、字段规则、接口返回、验收方式。
- 文档整体风格应保持“工程化、可执行、可验收”，不得写成宣传材料或概念方案。

## 7. 表格与代码块要求

- 表格用于描述模块清单、字段说明、接口说明、任务分工和验收指标。
- 表格字段名称应统一，例如“字段名、类型、是否必填、说明”。
- 代码块只放必要内容，例如数据库建表语句、JSON 示例、API 示例、配置文件示例。
- 代码块前后应说明用途，不得堆放无解释的大段代码。
- JSON、SQL、YAML 示例必须保证格式正确，字段命名保持一致。

## 8. 图示要求

- 技术方案中可加入：系统总体架构图、数据处理流程图、小智能体任务流转图、数据库核心表关系图、领导演示流程图。
- 图示风格应为黑白线框图，白底黑字，结构清晰，箭头方向明确。
- 图中不使用卡通图标、复杂渐变、装饰性背景。
- 图中文字应与正文术语保持一致，例如统一使用“CVE 采集智能体”“NVD 增强智能体”“本地模型处理智能体”。

## 9. Codex 执行要求

当 Codex 使用本规范处理文档时，应按以下顺序执行：

1. 先识别任务类型：生成新文档、清洗 Markdown、转换 DOCX、审查已有文档。
2. 如果输入是 Markdown，先执行 Markdown 残留清洗，再进行 Word 排版。
3. 如果输出 DOCX，必须检查正文中是否仍包含 `#`、`**`、反引号围栏、`---`、Markdown 表格管道符等残留。
4. 对长文档不要整篇打印到聊天界面，应生成文件或只输出问题清单、修改摘要和交付路径。
5. 不得擅自改变文档事实、技术路线、章节含义和项目边界。
6. 格式修改和内容改写应分开处理，避免因排版清洗导致正文含义变化。

## 10. 交付要求

- 文档交付最终包含：Markdown 源文件、Word 排版版本、PDF 只读版本。
- Markdown 文件用于版本管理和开发协作。
- Word 文件用于内部评审和领导汇报。
- PDF 文件用于正式发送和归档。
- 三个版本的标题、章节编号、表格编号和图编号应保持一致。
- 交付前必须进行一次残留检查，确认 DOCX 中无 Markdown 语法残留。
